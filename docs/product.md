# 解決したい課題 / 潜在的なニーズ

AIを使ってプログラミング学習しているが
- エラーが出たらAIにそのまま貼り付けて解決している
- 仮に動いたとしても「なぜ動いたか」説明できない
- チュートリアルが終わっても、自力で1からコードを書いて作ることができない
という状態になっている。

その結果として、
自分はエンジニアに向いてない、エンジニアやめたい、という不安を感じてしまう。

しかし、何が理解できているのかわからず、生活や学習期間、業務での技術要求の焦りから学習の進め方を修正できず、同じ状態を繰り返してしまう。

# 対象ユーザー（誰の、どんな状況か）

- エラーが出ると何も考えずにAIを使ってしまう人
- コードや課題を見ても内容を理解できず、焦りからAIを使ってしまう人
- 理解やスキルの積み上がりを感じられないまま進んでいる感覚がある人
- ポートフォリオを作ろうとしたら手が止まってしまう人

# プロダクトのカテゴリー

AI時代のプログラミング学習診断・改善アプリ

# 提供する価値（ユーザーが得られる変化・体験）

- 理解度把握：各スキル(アルゴリズム、Web技術、SQL、フレームワーク)の理解度、自力でどれくらいかけるのか把握する。
- AI使用について：AIに頼るべき場面と、自力で考えるべき場面の違いが分かり、自分で学習の進め方のコントロールができるようになる
- 学び直し：どこを学び直して、理解に時間をかけるべきかが分かる
- 気持ちの変化：AI依存で不安になっている状態から、自分の力で進められているという安心感を得られる
- エラー対処：エラーや課題がある時、まずは自分で考えるようになる

# 価値を実現するための最小限の機能（MVP）

- 診断：現状どれくらい理解できているか、初学者でも答えられるような技術的質問(基本アルゴリズム、データベース、Web技術、フレームワーク等)に10~20個程度答えてもらいます。
- 結果：診断内容をもとになぜその診断結果になったのか説明。
  - 例1：AI依存度「80%」：あなたは、自力でアルゴリズムを考えること、エラー解決をAIに任せる傾向が強いです。順次・選択・繰り返しの基本となるアルリズム、データの型を理解して、エラーも最初の10分程度は自力で考えて調べるようにしましょう。
  - 例2：AI依存度「20%」：あなたは、自力でアルゴリズムを考えて、エラー解決もほぼ自力で解決する傾向が強いです。今度はさらにAI活用を意識してコーディング効率を上げることを意識してみましょう。その際、AIの書いていることを理解して、いろんなコーディング方法を見つけて自分の引き出しを増やしていきましょう。
- アドバイス：診断内容に合わせてメッセージ
  - 例１：AI依存度が高い方へ：まずは、標準入力・標準出力を理解して、文字列操作、演算をしましょう。そしてif文で条件ごとに出力を変更したり、ループ文で繰り返し処理を理解してみましょう。この際、AIを使わずにまずは検索したり、公式ドキュメントを見ることを意識してみましょう。10分以上手が止まる場合、AIから答えでなくアドバイスをもらいましょう。
  - 例２：AI依存度が低い方へ：この調子でAIを適切なパートナーとして、効率的なコーディングやさらにロジックの引き出しを増やしていきましょう。
- 行動：診断結果に基づいて「次にやるべき行動」を提示し、ユーザーがその行動を実行・記録できる機能
  - AIとの向き合い方（例：エラーやわからないことがあっても10分は自力で考える）
  - 学習内容（例：for文の問題を1日1問、fizzbuzzを解く、配列のインデックス操作を練習する、デバッグ出力をする）
  - 期間（例：1週間）
  - 学習チェック（できた / できなかった）
- 履歴：診断結果と行動を保存できます。自身の理解度やAI依存度の変化を振り返ることができます。

ログインしてない状態では、診断と結果のみの利用として履歴に保存されません。
ログインしてる状態では、全ての機能を提供し、診断結果は履歴として参照することができます。

# 既存の代替手段・競合との違い

競合やユーザーの対処法として、
- Chat GPT などのAIツールを使い続ける
- Progate / Udemy などの教材を使い続ける
- Qiita / Zenn の記事を読み漁るが、記事の内容が古かったり、環境・条件の違いがあって解決できずにさらに記事を漁り続ける
- 何もわからないまま進めて挫折してしまう

ですが、本サービスは学習方法でなく、自分が何を理解できてないのかポイントを可視化して学習の方向性を正すことにフォーカスを当てています。

# 技術的にこだわりたい・工夫したい点（無理ない範囲で）

- 興味をそそるデザイン：ファーストビューはすぐにでも診断したくなるような好奇心をそそるようなUI/UXにしたいです。
- 質問内容：適切な質問内容を設計し、ユーザーの学習理解度、足りてない知識、立ち位置を正確に測ります。
- 矛盾のないDB設計、理解できるコード：理解負債を解消するためのアプリだからこそ、このサービスは私や他の方もわかるようなコードで実装するつもりです。

# なぜ自分がこれを作るのか（動機・原体験）
- 自分のエンジニア挫折経験から：過去にAIに頼ってエンジニアになったが、現場に入ってからプログラムがなぜ動くのかわからない状態に苦しみ、先輩に教えてもらっても理解できず強い焦りを感じました。
この経験から、同じようにAI依存で苦しむ人を増やさず、一度立ち止まって学習方法を見つめ直すという選択を与えたいです。
